## Project architecture

- **React 18 SPA** with Vite 8, TypeScript, Tailwind CSS, Shadcn/ui.
- **State**: AppContext (`src/contexts/AppContext.tsx`) is the main data hub — CRUD, auth, Realtime subscriptions, and offline cache.
- **TanStack Query**: used only in `src/features/presupuestos/hooks/usePresupuestos.ts` (`useQuery`/`useMutation`). Everything else flows through AppContext + Supabase directly.
- **Supabase** (`@supabase/supabase-js` v2) for auth, DB, and Realtime.
- **Path alias**: `@/` → `src/` (vite.config.ts + vitest.config.ts).
- **PWA**: `vite-plugin-pwa` with `registerType: "autoUpdate"`. Generates `dist/manifest.webmanifest` and `dist/sw.js` on build. Dev mode PWA is also enabled.
- **Vite uses rolldown** (vite 8): build output uses `rolldown-runtime` chunk instead of rollup.
- **Entrypoint chain**: `src/main.tsx` → `App.tsx` → `pages/Index.tsx` → `<AppProvider> <AppLayout />`.
- **Features dir**: `src/features/{clientes,financiero,presupuestos,proyectos}/`, each with own `components/` and `hooks/`.
- **Services dir**: `src/services/` (CalculoService, PresupuestoService, RenglonesService, ExportService, offline).
- **Hooks dir**: `src/hooks/` has 17 hooks (useChangeOrders, useChecklistCalidad, useEquipos, etc.). Feature-specific hooks may duplicate names — check imports.
- **No CI workflows** (only dependabot for devcontainers). No automated PR checks besides local commands.
- **`dev-dist/`** is auto-generated by vite-plugin-pwa in dev mode. Listed in `.gitignore`.

## Supabase — critical facts

### Env vars (required)
```
VITE_SUPABASE_URL=<url>
VITE_SUPABASE_ANON_KEY=<key>
```
`.env` has real creds (live Supabase project); `.env.example` has placeholders. **Do not commit `.env`.**

### 28 SQL files — only ONE matters for schema
Use **`SYNC_SUPABASE_FINAL.sql`** as the authoritative schema (17 tables, fully idempotent). Ignore all other `.sql` files EXCEPT:
- `scripts/fix-rls-recursion.sql` — must be run against the database to fix RLS infinite recursion (42P17)
- `migration_equipos.sql` contains `presupuestos_select_team` which is NOT in SYNC_SUPABASE_FINAL.sql and causes RLS recursion cascade

### 17 tables
`clientes`, `proyectos`, `presupuestos`, `transacciones`, `actividades`, `equipos`, `equipo_miembros`, `renglones`, `renglon_usage`, `renglon_precios_historial`, `cambios_presupuesto`, `materiales_proyecto`, `movimientos_materiales`, `conciliaciones`, `partidas_conciliacion`, `checklist_items`, `notificaciones`

### `transacciones.proyecto_id` type
Must stay `text` (not `uuid`) — the app writes `'admin'` as proyecto_id for office transactions.

### 4 feature modules with Supabase hooks
| Module | Hook file | Tables used |
|---|---|---|
| Change Orders | `src/hooks/useChangeOrders.ts` | `cambios_presupuesto` |
| Materials tracking | `src/hooks/useTrazabilidadMateriales.ts` | `materiales_proyecto`, `movimientos_materiales` |
| Bank reconciliation | `src/hooks/useConciliacionBancaria.ts` | `conciliaciones`, `partidas_conciliacion` |
| Quality checklists | `src/hooks/useChecklistCalidad.ts` | `checklist_items` |

### RLS — known issue: infinite recursion (42P17)
**The database has conflicting RLS policies that cause circular recursion between `equipos` and `equipo_miembros`**, cascading to `presupuestos` via `presupuestos_select_team`. Multiple migration scripts applied different policy names over time, leaving a mix.

To fix, run **`scripts/fix-rls-recursion.sql`** in Supabase SQL Editor. It:
1. Creates `public.user_owns_equipo()` (SECURITY DEFINER) to break the cycle
2. Drops ALL known policy names on `equipos`, `equipo_miembros`, and `presupuestos` (handles names from all the different SQL files)
3. Recreates with simple owner-only or SECURITY DEFINER-based policies

If the error persists after running the script, verify via `SELECT * FROM pg_policies WHERE tablename IN ('equipos','equipo_miembros','presupuestos')` that no cross-table subqueries remain.

### `costo_directo` on `presupuestos`
Column already exists in `SYNC_SUPABASE_FINAL.sql`. `PresupuestoService.ts:191` writes it. `PresupuestoScreen.handleSave()` must include it in the payload (was hardcoded to 0 — already fixed).

## Offline-first system

Built into AppContext via `src/services/offline.ts`.

- **Cache**: every successful Supabase fetch saves to `localStorage` under key `offline_{table}_{userId}`. On fetch failure (offline or 500), each table loads independently from cache — one broken table never blocks others.
- **Pending queue**: mutations made while offline are saved as `PendingMutation` records in `localStorage` under `offline_pending_{userId}`. State updates are applied optimistically (immediate UI + cache save).
- **Auto-sync**: on `online` event, pending mutations replay in order against Supabase. After all sync, fresh data is fetched. Sync happens automatically via `useEffect` on `[isOnline, session?.user.id]`.
- **Edge case**: if the Supabase project lacks some tables (500), those tables load from cache or stay empty.

## Config quirks

- **eslint**: `no-unused-vars` OFF, `no-explicit-any` OFF, `react-refresh/only-export-components` OFF.
- **Build chunks**: vendor-react, vendor-charts (recharts), vendor-icons (lucide), vendor-forms (zod/hook-form), vendor-other.
- **vitest**: jsdom environment, `@` alias, `./src/setupTests.ts` (imports `@testing-library/jest-dom`).
- **TypeScript strict**: `tsconfig.app.json` has `strict: true`, `noUnusedLocals`, `noUnusedParameters`, `noImplicitAny` — so `npm run typecheck` catches things eslint ignores.
- **Tests**: 3 test files in `src/utils/`. No e2e tests.
- **Vercel**: SPA catch-all rewrite `/(.*)` → `/index.html` in `vercel.json`. Headers for `manifest.webmanifest` (Content-Type: `application/manifest+json`) to prevent 401 in preview deployments with Deployment Protection.
